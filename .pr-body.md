## Summary

PR #297 (closes #296) shipped `--no-prefix-user-supplied-names` as an opt-in flag that drops the `${stackName}-` prefix from user-supplied physical names on Pattern B resource types (IAM Role / User / Group / InstanceProfile / ELBv2 LoadBalancer / TargetGroup). Flipping the flag on against a stack already deployed under the legacy convention causes cdkd's existing diff-driven REPLACEMENT path to silently destroy and re-create every affected resource — the AWS-deployed name is the prefixed `MyStack-my-role` but the new template intent is the unprefixed `my-role`, so the diff classifies the name change as an immutable property change. Users discover this only mid-deploy.

This PR adds a deploy-time pre-flight migration check. Before any provider call, `cdkd deploy` walks the existing state and flags every Pattern B resource whose `physicalId` still starts with `${stackName}-`, lists each affected resource, and prompts the user before proceeding. The prompt defaults to **no** (the destructive side effect requires explicit `y`, mirroring `cdkd destroy --remove-protection`'s flipped default) and is skipped under `-y` / `--yes` for CI.

Example output:

```text
WARNING: --no-prefix-user-supplied-names will REPLACE 2 resource(s) whose
AWS physical name is still prefixed with the stack name:
  - MyRole (AWS::IAM::Role): MyStack-my-role -> my-role
  - MyLb (AWS::ElasticLoadBalancingV2::LoadBalancer): MyStack-my-lb -> my-lb
These resources will be REPLACED because the new naming convention drops
the stack-name prefix.

Continue? (y/N):
```

The check is a no-op when (a) the flag is off, (b) state is empty (first-time deploy), or (c) every Pattern B resource is already unprefixed (the stack was originally deployed with the flag on). Declining the prompt exits cleanly with a "no resources modified" message — nothing has been touched yet.

## Changes

- **New** `PATTERN_B_RESOURCE_TYPES` exported from [src/provisioning/resource-name.ts](src/provisioning/resource-name.ts) — single source of truth for the type set the flag affects (kept in sync with the Pattern B provider call sites that consume `generateResourceNameWithFallback`).
- **New** [src/cli/commands/prefix-migration-check.ts](src/cli/commands/prefix-migration-check.ts) — pure-functional `findPendingPrefixRenames(stackName, state)` helper + interactive `promptMigrationConfirm(renames, {yes})`. Each pending rename is reported as `{ logicalId, resourceType, oldPhysicalId, newPhysicalId }` so callers can render an exact preview of what will replace.
- **Wired** the check into `runStackInner` in [src/cli/commands/deploy.ts](src/cli/commands/deploy.ts) — fires inside the per-stack `try { ... } finally { ... }` block (after state load, before `new DeployEngine(...)`) so per-stack AWS-client teardown runs whether the user confirms or declines.
- **Docs** new H3 subsection under the existing `--no-prefix-user-supplied-names` section in [docs/cli-reference.md](docs/cli-reference.md).
