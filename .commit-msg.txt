feat(deploy): pre-flight warning when --no-prefix-user-supplied-names triggers REPLACEMENT

PR #297 (issue #296) shipped `--no-prefix-user-supplied-names` as an
opt-in flag that drops the `${stackName}-` prefix from user-supplied
physical names on Pattern B resource types (IAM Role / User / Group /
InstanceProfile / ELBv2 LoadBalancer / TargetGroup). Flipping the flag
on against a stack already deployed under the legacy convention causes
cdkd's existing diff-driven REPLACEMENT path to silently destroy and
re-create every affected resource — the AWS-deployed name is the
prefixed `MyStack-my-role` but the new template intent is the
unprefixed `my-role`, so the diff classifies the name change as an
immutable property change. Users discover this only mid-deploy.

This PR adds a deploy-time pre-flight migration check (issue #300).
Before any provider call, `cdkd deploy` walks the existing state and
flags every Pattern B resource whose `physicalId` still starts with
`${stackName}-`. Each affected resource is listed
(`<logicalId> (<resourceType>): <oldPhysicalId> -> <newPhysicalId>`)
and the user is prompted `Continue? (y/N)` (default no; -y / --yes
skips the prompt). Declining exits cleanly with a "no resources
modified" message — nothing has been touched yet.

The check is a no-op when (a) the flag is off, (b) state is empty
(first-time deploy), or (c) every Pattern B resource is already
unprefixed (the stack was originally deployed with the flag on).

Implementation:
- src/provisioning/resource-name.ts: new exported
  PATTERN_B_RESOURCE_TYPES list (single source of truth for the
  type set the flag affects).
- src/cli/commands/prefix-migration-check.ts: new module with the
  pure-functional findPendingPrefixRenames(stackName, state) helper
  and the interactive promptMigrationConfirm(renames, {yes}) prompt.
- src/cli/commands/deploy.ts: wires the check into runStackInner's
  per-stack try / finally block — fires after state load and before
  `new DeployEngine(...)` so per-stack AWS-client teardown still
  runs whether the user confirms or declines.

13 new unit tests in tests/unit/cli/prefix-migration-check.test.ts
cover no-state / no Pattern B / already-unprefixed / mixed Pattern A+B
/ every Pattern B type / supplied-stackName-not-state-stackName +
prompt yes/no/default/empty/mixed-case/close-on-confirm. Docs added
in docs/cli-reference.md under the existing --no-prefix-user-supplied-names
section.

Closes #300
