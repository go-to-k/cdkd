fix(deploy): non-TTY guard + empty-suffix skip in prefix-migration pre-flight

PR review (1-reviewer dispatched by /review-pr 304) flagged:

1. (MAJOR) promptMigrationConfirm hangs on closed stdin / non-TTY
   environments without --yes — CI without the flag would silently
   decline + exit 0. Fix: explicit isTTY guard that throws an
   actionable error pointing at --yes; readline is NOT opened.
   opts.yes short-circuits the guard so CI happy path is preserved.

2. (MINOR) findPendingPrefixRenames: skip entries whose physicalId
   equals `${stackName}-` exactly (newPhysicalId would be `''`), so
   no `to ""` rename line surfaces.

Tests: 2 new + 4 existing prompt tests updated to pretendTTY() (the
existing tests assumed default vitest TTY=true, which it is NOT in
CI — they would have broken under the new guard). All 16 pass; full
suite 3143 pass.

The state-load-before-lock race the reviewer flagged as "major" is
deliberately kept as documented best-effort UX: the actual deploy
still re-reads state under the lock, so the worst case is a stale
prompt snapshot — not data corruption. Promoting to a strict-lock
design would require threading skipPrefix into DeployEngine; out of
scope for this PR.
